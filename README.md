# Lex-Ask — EU AI Act RAG

Ask questions about the EU AI Act and get grounded, cited answers in seconds.

**Live demo:** [lex-ask.jhutty.de](https://lex-ask.jhutty.de)

## What it does

Lex-Ask is a Retrieval-Augmented Generation (RAG) app that answers natural-language questions about the EU AI Act (Regulation (EU) 2024/1689). 

Answers are grounded in the regulation text with instructions to cite sources and refuse when context is insufficient.

## Quick start

### Prerequisites

- Python 3.14+
- A Gemini API key

### Setup

```bash
git clone https://github.com/jhutty/lex-ask
cd lex-ask
pip install -r requirements.txt

# Set your API key
export GEMINI_KEY="your_key_here"

# Fetch, parse, embed — run once to populate the vector store
python app.py --prep

# Start the app
python app.py
```

Open `http://localhost:5001` and start asking questions.

## How it works

1. **Data ingestion** — The full EU AI Act XML is fetched from EUR-Lex and parsed into 113 articles, 13 annexes, and a metadata overview chunk.

2. **Embedding** — Each chunk is embedded using `gemini-embedding-2` and stored in a persistent local ChromaDB collection. Chunks are keyed by article/annex ID, so re-runs skip already-stored items.

3. **Retrieval** — When a user asks a question, the query is embedded with the same model. ChromaDB returns the nearest chunks by cosine distance. A distance threshold (0.7) filters out low-relevance results. Not all top-k hits are useful, and surfacing bad context degrades answer quality.

4. **Generation** — Retrieved chunks are injected into a system prompt that instructs Gemini Flash Lite to answer *only* from the provided context and cite specific article numbers. If no chunks pass the threshold, the app returns a refusal rather than guessing.

5. **Frontend** — A minimal FastHTML interface with HTMX-powered form submission. The response shows the answer, source articles, and distance scores for transparency.

## Architecture

```
User question
     │
     ▼
FastHTML frontend
     │
     ▼
Embed query (Gemini)  ──►  ChromaDB  ──►  Top-k chunks (distance-filtered)
                                                │
                                                ▼
                                          LLM generation (Gemini Flash Lite)
                                                │
                                                ▼
                                          Grounded answer + cited articles
```

**Why these choices:**
- **ChromaDB (embedded)** — no separate database server, zero infrastructure cost, ships inside the container
- **Gemini free tier** — both embeddings and generation at no cost, sufficient for a low-traffic demo
- **FastHTML** — lightweight Python-native web framework, no JS build step, keeps the stack minimal

## Deployment

The app runs on Azure Container Apps. The key steps:

```bash
# 1. Build and push the Docker image
docker build -t jhutty/lex-ask:latest .
docker push jhutty/lex-ask:latest

# 2. Create Azure resources
az group create --name lex-ask --location northeurope
az containerapp env create --name lex-ask-env --resource-group lex-ask --location northeurope

# 3. Deploy - set the secret and env var at creation time to avoid crash-on-start
az containerapp create \
  --name lex-ask \
  --resource-group lex-ask \
  --environment lex-ask-env \
  --image jhutty/lex-ask:latest \
  --target-port 5001 \
  --ingress external \
  --secrets gemini-key="$GEMINI_KEY" \
  --env-vars GEMINI_KEY=secretref:gemini-key

# 4. Custom domain (requires CNAME pointing to the Azure-assigned FQDN first)
az containerapp hostname add --name lex-ask --resource-group lex-ask --hostname lex-ask.jhutty.de
az containerapp hostname bind --name lex-ask --resource-group lex-ask --hostname lex-ask.jhutty.de \
  --environment lex-ask-env --validation-method http
```

**Note:** The `chroma_db/` directory is baked into the Docker image at build time, so the `--prep` step must complete locally before `docker build`.

## Design decisions

- **Distance threshold over raw top-k:** Not all top-k results are relevant. Without filtering by similarity score, low-quality chunks pollute the context and the LLM confidently generates wrong answers from them. A threshold of 0.7 strikes a balance between recall and precision.
- **Strict grounding via system prompt:** The generation prompt explicitly forbids the LLM from using training data and requires article citations. Without this, the model supplements with its own knowledge and invents plausible but incorrect references.
- **Embedded vector store:** ChromaDB runs in-process with no separate server, keeping the deployment to a single container with zero infrastructure overhead.
- **Baked-in data:** The `chroma_db/` directory is built into the Docker image at build time rather than populated at runtime. This avoids startup latency and API calls on every container restart.

## Limitations & future work

**Current limitations:**
- No sub-article chunking. Long articles are embedded as single chunks, which can dilute retrieval precision
- No evaluation suite yet (answer quality is verified manually)
- Single-language (English). The Act is available in all EU languages

**Planned for V2:**
- [ ] Automated eval suite (Q&A pairs to measure retrieval precision and answer quality)
- [ ] Sub-article chunking with overlap for better retrieval on detailed questions
- [ ] CI/CD pipeline via GitHub Actions
- [ ] Support for additional EU legislation
